    Bolt Cowork v0.2.6 — Manuel Test Checklist

    Her maddeyi terminalde dene, sonucu ✅ / ❌ olarak işaretle. Test ortamı: boş bir klasörde bolt-cowork --dir ./test-workspace ile başla.

    1. CLI Başlangıç

    • [ ] bolt-cowork --help → flagler ve açıklamalar görünüyor
    • [ ] bolt-cowork --version → sürüm bilgisini config/LLM yüklemeden gösteriyor
    • [ ] bolt-cowork --dir ./test-workspace → REPL açılıyor, banner (ASCII-only) görünüyor
    • [ ] bolt-cowork --dir /nonexistent → anlamlı hata mesajı, crash yok
    • [ ] bolt-cowork --provider nonexistent → provider bulunamadı hatası
    • [ ] bolt-cowork --approval invalid → geçersiz approval mode hatası

    2. Config Yükleme

    • [ ] ~/.bolt-cowork/config.yaml yokken başlat → varsayılan config ile çalışıyor
    • [ ] config.yaml içinde env var kullan (${API_KEY}) → genişletiliyor
    • [ ] Geçersiz YAML → açıklayıcı parse hatası, panic yok
    • [ ] Tilde (~) path genişletmesi çalışıyor (ör. skills_dir: ~/my-skills)

    3. REPL Temel Komutlar

    • [ ] Boş input (sadece Enter) → hata yok, yeni prompt
    • [ ] /help → komut listesi görünüyor
    • [ ] /dir → mevcut çalışma dizinini gösteriyor
    • [ ] /dir ./subdir → dizin değişiyor (subdir varsa)
    • [ ] /dir /outside/sandbox → sandbox dışı, reddediliyor
    • [ ] /dir - → önceki dizine dönüyor
    • [ ] /quit veya Ctrl+C → temiz çıkış, "goodbye" mesajı
    • [ ] Bilinmeyen komut (/asdf) → "unknown command" hatası

    4. Agent Loop — Dosya İşlemleri

    • [ ] "create a file called hello.txt with content 'merhaba'" → dosya oluşuyor, içerik doğru
    • [ ] "read hello.txt" → içerik gösteriliyor
    • [ ] "rename hello.txt to selam.txt" → isim değişiyor
    • [ ] "copy selam.txt to backup.txt" → kopya oluşuyor
    • [ ] "delete backup.txt" → onay isteniyor (dangerous), onaylayınca siliniyor
    • [ ] "list files" → mevcut dosyalar listeleniyor
    • [ ] "create directory subdir" → klasör oluşuyor

    5. Approval Gate (Onay Kapısı)

    5a. Full Approval Mode (--approval full)

    • [ ] Her adımda onay isteniyor (plan + execute)
    • [ ] Reject yapınca → işlem iptal, dosya değişmemiş
    • [ ] ApproveAll seçince → kalan adımlar otomatik onaylanıyor

    5b. Dangerous-Only Mode (--approval dangerous-only)

    • [ ] read/list gibi güvenli işlemler → otomatik onay, soru yok
    • [ ] delete/overwrite → onay isteniyor, [DANGEROUS] etiketi + neden görünüyor
    • [ ] ApproveAll seçtikten sonra başka bir dangerous adım gelince → YİNE onay istiyor
    • [ ] Non-dangerous adımlar ApproveAll sonrası → otomatik geçiyor

    5c. None Mode (--approval none)

    • [ ] Hiçbir adımda onay istemiyor
    • [ ] Dangerous işlemler bile direkt çalışıyor (uyarı: bu mod tehlikeli)

    6. Protected Path Güvenliği

    Bu testleri --approval none ile yap ki approval araya girmesin.

    • [ ] "read .env.local" → "protected file" hatası
    • [ ] "read .bolt-cowork/config.yaml" → reddediliyor
    • [ ] "write to .ssh/test" → reddediliyor
    • [ ] "copy hello.txt to .ssh/authorized_keys" → reddediliyor
    • [ ] "delete .gnupg/something" → reddediliyor
    • [ ] "mkdir .ssh/newdir" → reddediliyor
    • [ ] "list .ssh" → reddediliyor (dizin root'u da korumalı)

    6a. Symlink Bypass Denemeleri (Linux/Mac)

    • [ ] ln -s .env.local safe.txt → "read safe.txt" → reddediliyor
    • [ ] ln -s ../.ssh safe_dir → "mkdir safe_dir/test" → reddediliyor
    • [ ] ln -s ../.ssh safe_dir → "copy file.txt to safe_dir" → reddediliyor

    7. Secret Redaction

    • [ ] Config'e bilinen bir API key yaz (ör. sk-test-12345)
    • [ ] LLM hata verirse → hata mesajında API key [REDACTED] olarak görünüyor
    • [ ] Step result'ta API key geçiyorsa → [REDACTED]
    • [ ] Overlapping keys testi: sk-test ve sk-test-abcdef varsa → sk-test-abcdef tam redact ediliyor, parçalanmıyor

    8. Provider / Fallback Chain

    • [ ] Geçerli API key ile → LLM yanıt veriyor
    • [ ] Geçersiz API key ile → anlamlı hata, API key redact edilmiş
    • [ ] Birden fazla provider tanımlı, ilki başarısız → ikinciye düşüyor (fallback)
    • [ ] Tüm provider'lar başarısız → "all providers failed" hatası

    9. Skill Sistemi

    9a. Skill Yükleme

    • [ ] ./bolt-skills/ içine bir SKILL.md dosyası koy → başlangıçta yükleniyor
    • [ ] ~/.bolt-cowork/skills/ içine global skill koy → yükleniyor
    • [ ] Aynı isimde skill her iki yerde → project olanı kazanıyor, uyarı loglanıyor
    • [ ] Geçersiz YAML frontmatter → uyarı, fallback değerlerle yükleniyor
    • [ ] Boş dosya → isim dosya adından türetiliyor, hata yok
    • [ ] CRLF satır sonlu SKILL.md → düzgün parse ediliyor

    9b. Skill Tetikleme

    • [ ] /skill-adı komutu → ilgili skill aktif oluyor
    • [ ] Bilinmeyen skill adı (/nonexistent-skill) → "skill not found" hatası
    • [ ] auto_trigger: true olan skill → eşleşen komutta otomatik aktif

    9c. Skill Context Injection

    • [ ] Aktif skill varken LLM çağrılınca → system prompt'ta skill içeriği görünüyor (debug/verbose modda kontrol et)

    10. Context Trimming

    • [ ] 20'den fazla mesaj gönder → eski mesajlar kırpılıyor, son 20 kalıyor
    • [ ] Çok uzun tek mesaj (>32K karakter) → trimming çalışıyor, crash yok
    • [ ] Kırpma sonrası summary mesajı ekleniyor

    11. Error Handling — Edge Cases

    • [ ] LLM boş yanıt dönerse → crash yok, anlamlı hata
    • [ ] LLM malformed JSON dönerse → "invalid action" hatası, devam edebiliyor
    • [ ] Disk dolu simülasyonu (yazma başarısız) → hata mesajı, crash yok
    • [ ] Ctrl+C tam işlem sırasında → graceful shutdown, yarım dosya bırakmıyor

    12. Hata Mesajı Stili

    • [ ] Tüm hata mesajları küçük harfle başlıyor (büyük harf yok)
    • [ ] Hata mesajlarının sonunda nokta yok
    • [ ] [DANGEROUS] etiketinde neden açıklaması var (ör. "permanently removes file")

    13. Windows Uyumluluğu (Windows'ta test ediyorsan)

    • [ ] Banner ASCII-only, kutu karakterleri yok
    • [ ] Symlink testleri skip ediliyor (panic yok)
    • [ ] Path separator \ düzgün çalışıyor
    • [ ] Renklendirme düzgün görünüyor veya graceful degradation

    14. Build & Lint Doğrulaması

    go build ./...          # exit code 0
go vet ./...            # temiz
go test ./...           # hepsi PASS
gofmt -l ./...          # boş çıktı (formatlanmamış dosya yok)
go mod tidy && git diff # değişiklik yok

    15. Codex Ek Kontrolleri

    Bu bölüm Claude checklist'ine ek olarak Codex'in aklına gelen güvenlik, kalite ve stabilizasyon maddeleridir. Gemini ekleme yaptıktan sonra bu dosya tek kaynak checklist kabul edilecek.

    15a. Test İzolasyonu ve Güvenli Çalışma Alanı

    • [ ] Testler gerçek ~/Documents, ~/Desktop, ~/Downloads veya kullanıcı home dizininde dosya oluşturup silmiyor
    • [ ] Testlerde os.UserHomeDir() veya os.Getenv("HOME") gerçek home'a erişmek için kullanılmıyor
    • [ ] Dosya operasyon testleri yalnızca testdata/, proje altındaki test workspace veya t.TempDir() kullanıyor
    • [ ] Sandbox testleri gerçek sistem dizinlerine dokunmadan dış erişimi reddettiğini kanıtlıyor
    • [ ] Manuel test workspace'i proje dizini altında izole oluşturuluyor ve test sonunda güvenle temizlenebiliyor

    15b. Sandbox ve Path Bypass Denemeleri

    • [ ] ../ traversal denemeleri sandbox dışına çıkamıyor
    • [ ] Mutlak path ile sandbox dışı dosya okuma/yazma reddediliyor
    • [ ] Windows path separator varyantları (\, /, karışık kullanım) sandbox kontrolünü bypass etmiyor
    • [ ] Case-insensitive protected path denemeleri (.SSH, .Env.Local) Windows'ta reddediliyor
    • [ ] Hedef path temizleme/canonicalization sonrası kontrol ediliyor, ham input'a güvenilmiyor
    • [ ] Symlink desteklenen sistemlerde symlink hedefi protected path ise işlem reddediliyor

    15c. Approval Mode Semantiği

    • [ ] --approval full skill approval dahil tüm approval gate'leri gösteriyor
    • [ ] --approval plan-only skill approval göstermiyor, yalnızca plan aşamasında duruyor
    • [ ] --approval dangerous-only dokümantasyonla uyumlu çalışıyor; REPL kısayolu /mode build, CLI alias olarak --approval dangerous beklenmiyor
    • [ ] --approval none hiçbir approval gate göstermiyor, fakat sandbox/protected path güvenliği devam ediyor
    • [ ] ApproveAll dangerous adımları kalıcı olarak bypass etmiyor
    • [ ] Plan revise akışı en fazla 3 revizyonla sınırlı
    • [ ] Result aşamasında Accept / Rollback seçenekleri beklenen dosya durumunu üretiyor

    15d. Skill Sistemi Ekleri

    • [ ] Skill matching case-insensitive çalışıyor
    • [ ] SetForceSkills() veya manuel force edilen skill listesi her Run() çağrısından sonra temizleniyor
    • [ ] /skills, /skill <name> ve /use <name> komutları dokümantasyonla aynı davranıyor
    • [ ] Project skill, global skill ve bundled skill öncelik sırası tutarlı
    • [ ] Aynı isimli skill çakışmasında log mesajı anlaşılır ve secret/path sızdırmıyor
    • [ ] Embedded bundled skills release binary içinde erişilebilir
    • [ ] Invalid/empty/CRLF SKILL.md dosyaları panic olmadan fallback davranışı gösteriyor

    15e. Provider, Model Switching ve Terminoloji

    • [ ] /model komutu model adına göre provider'ı doğru auto-detect ediyor
    • [ ] Model switch sonrası conversation history beklenen şekilde korunuyor veya dokümante edildiği gibi davranıyor
    • [ ] Provider fallback hatalarında hangi provider'ın başarısız olduğu anlaşılır ama API key redact edilmiş
    • [ ] Runtime provider kavramı Claude Code / Codex / Gemini CLI gibi geliştirme araçlarıyla karıştırılmıyor
    • [ ] OpenAI, Anthropic ve Gemini runtime provider dokümantasyonu gerçek flag/config isimleriyle uyumlu

    15f. Secret Redaction Ekleri

    • [ ] Redaction uzun secret'ları kısa secret'lardan önce işliyor (sk-test-abcdef parçalanmıyor)
    • [ ] Boş string veya çok kısa değerler tüm çıktıyı yanlışlıkla redact etmiyor
    • [ ] Error wrapping redaction'ı geri almıyor veya secret'ı yeniden loglamıyor
    • [ ] Step result, provider error, config error ve debug output aynı redaction politikasını kullanıyor
    • [ ] Test secret'ları gerçek API key formatına benzese bile test-only değerlerden oluşuyor

    15g. Error Handling ve Mesaj Stili

    • [ ] Kullanıcıya görünen hata mesajları küçük harfle başlıyor ve nokta ile bitmiyor
    • [ ] Internal error'lar fmt.Errorf("context: %w", err) ile bağlam ekleyerek wrap ediliyor
    • [ ] Panic yerine anlamlı hata dönen edge-case testleri var
    • [ ] Malformed LLM action output'u agent loop'u kilitlemiyor
    • [ ] Ctrl+C veya context cancellation yarım dosya bırakmıyor
    • [ ] Invalid path, permission denied ve disk write failure mesajları kullanıcıya ne olduğunu anlatıyor

    15h. REPL ve Conversation History Ekleri

    • [ ] Boş input history'ye anlamlı olmayan mesaj olarak eklenmiyor
    • [ ] /clear conversation history'yi temizliyor
    • [ ] 20-turn FIFO cap testinde son mesajlar korunuyor
    • [ ] Çok uzun tek mesaj trimming sonrası crash üretmiyor
    • [ ] Readline history sensitive input veya API key saklamıyor
    • [ ] Unknown slash command ile unknown skill ayrımı kullanıcıya net gösteriliyor

    • [ ] Shell script'ler ShellCheck ile uyumlu veya bilinen istisnalar dokümante

    • [ ] go mod tidy sonrası sadece beklenen modül değişiklikleri var veya hiç diff yok

    17. Gemini Ek Kontrolleri - Mimari ve Derinlemesine Güvenlik

    Bu bölüm projenin senior developer ve code reviewer perspektifinden, GEMINI.md'deki katı kurallara ve Go dilinin çalışma karakteristiğine odaklanan maddelerdir.

    17a. Sinyal Yönetimi ve Kaynak Temizliği (Interruption Handling)

    • [ ] Uzun süren bir plan oluşturma (LLM stream) sırasında Ctrl+C basıldığında uygulama anında kapanıyor mu yoksa "aborting..." mesajıyla temiz çıkış mı yapıyor?
    • [ ] Bir dosya yazma (write) işlemi sırasında Ctrl+C basılırsa, yarım/bozuk dosya kalıyor mu? (Atomic write kontrolü)
    • [ ] Birden fazla Ctrl+C basıldığında uygulamanın "force quit" davranışı tutarlı mı?

    17b. Plan Revizyon Döngüsü ve Durum Yönetimi (State Consistency)

    • [ ] Kullanıcı bir planı reddedip geri bildirim verdiğinde, LLM'e giden yeni prompt'ta önceki reddedilen plan ve kullanıcı yorumu net bir şekilde yer alıyor mu?
    • [ ] 3 revizyon limitine ulaşıldığında uygulama kullanıcıyı nasıl yönlendiriyor? (Sessizce durmamalı, bir sonraki adımı önermeli)
    • [ ] Plan revizyonu sırasında /dir veya /model gibi komutlar çalıştırıldığında planlama süreci bozuluyor mu?

    17c. Sandbox ve Path Traversal (Uç Durumlar)

    • [ ] ./test/../test/hello.txt gibi "redundant" ama geçerli path'ler sandbox içinde doğru çözülüyor mu?
    • [ ] Windows'ta CON, PRN, AUX, NUL gibi rezerve edilmiş dosya isimlerine yazma denemeleri engelleniyor mu?
    • [ ] allowed_dirs içinde yer alan bir dizin silindikten sonra, agent bu dizini otomatik tekrar oluşturmaya çalışıyor mu yoksa hata mı veriyor?

    17d. Skill Metadata ve Frontmatter Doğruluğu

    • [ ] Bir SKILL.md dosyasında frontmatter (--- blokları) bozuksa veya eksikse, skill sistemi dosyayı tamamen mi reddediyor yoksa sadece bozuk kısmı mı yoksayıyor?
    • [ ] SkillScope enum değerleri (Project, Global, Bundled) dışında bir değer verildiğinde sistemin fallback davranışı nedir?
    • [ ] Skill içindeki description alanı /skills listesinde doğru ve kırpılmadan görünüyor mu?

    17e. Levenshtein ve Kullanıcı Yardımı (UI/UX)

    • [ ] Kullanıcı /hellp (hatalı help) yazdığında Levenshtein algoritması /help önerisini "Did you mean...?" şeklinde gösteriyor mu?
    • [ ] /use komutunda skill ismi hatalıysa benzer skill isimleri öneriliyor mu?
    • [ ] Boşluk karakteri içeren skill isimleri veya path'ler tırnak içinde doğru şekilde ele alınıyor mu?

    17f. Gelişmiş Redaction ve Log Güvenliği

    • [ ] Çok satırlı (multi-line) bir error mesajı içinde secret varsa, redaction tüm satırlarda çalışıyor mu?
    • [ ] Log dosyası tutuluyorsa (eğer aktifse), /key set komutuyla girilen anahtar log dosyasına düz metin olarak düşüyor mu?
    • [ ] LLM'e gönderilen sistem prompt'u içinde yanlışlıkla konfigürasyondaki secret'lar yer alıyor mu? (Debug modda kontrol)

    17g. Dokümantasyon ve Gerçeklik Kontrolü (Single Source of Truth)

    • [ ] bolt-cowork --version çıktısı GEMINI.md veya go.mod'daki sürümle (v0.2.6) birebir eşleşiyor mu?
    • [ ] README.md'deki örnek senaryolar (quickstart) güncel CLI flag'leri ile hatasız çalışıyor mu?
    • [ ] /help çıktısındaki tüm komutlar internal/tool/builtins.go içindeki liste ile senkronize mi?

    17h. Skill & Context Etkileşimi (Lifecycle)

    • [ ] 20 mesajlık FIFO sınırı dolup mesajlar kırpıldığında (trimming), aktif olan skill'ler sistem prompt'unda kalmaya devam ediyor mu? (Skill'ler context'ten bağımsız yaşamalı)
    • [ ] /clear komutu çalıştırıldığında aktif olan skill'ler de deaktif oluyor mu? (Beklenen: history temizlenmeli ama skill aktif kalmalı)

    17i. Terminal ve Spinner (UX)

    • [ ] Windows Terminal'de (veya CMD/PowerShell) işlem sürerken (spinner aktifken) pencere boyutu değiştirildiğinde çıktı bozuluyor mu?
    • [ ] LLM'den yanıt beklerken (spinner dönerken) bir hata oluşursa, spinner düzgünce durup hata mesajını temiz bir satıra mı yazıyor?

    17j. Gelişmiş Redaction (Overkill Testi)

    • [ ] Konfigürasyonda çok kısa bir değer varsa (örneğin: id: 12), redaction sistemi çıktıdaki her "12" sayısını mı siliyor? (Minimum karakter sınırı kontrolü: en az 5-6 karakterden kısa olanlar redact edilmemeli)

    17k. Path Normalization (Windows Spesifik)

    • [ ] allowed_dirs listesine C:/data (forward slash) eklenmişken, agent'a C:\data (back slash) üzerinden bir dosya yazdırıldığında sandbox bunu doğru eşleştiriyor mu?
    • [ ] Case-insensitive disklerde (Windows) C:\Data ve C:\data arasındaki fark sandbox bypass'ına neden oluyor mu?
